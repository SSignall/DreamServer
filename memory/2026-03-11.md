# 2026-03-11

## Security Review: timingSafeEqual Length Oracle Attack

**Context:** Android-16 attempted to "fix" timingSafeEqual by adding an early length check.

**The Vulnerability:**
```javascript
// ❌ WRONG - Creates length oracle attack
if (a.length > MAX_LEN || b.length > MAX_LEN) return false;
```
An attacker can measure response times to determine the exact length threshold (fast response = early return, slow response = loop executed).

**The Correct Pattern (commit 697215a7):**
```javascript
function timingSafeEqual(a, b) {
  if (typeof a !== 'string' || typeof b !== 'string') return false;
  const MAX_LEN = 256;
  
  let result = 0;
  // Always iterate MAX_LEN - same time regardless of input length
  for (let i = 0; i < MAX_LEN; i++) {
    const charA = i < a.length ? a.charCodeAt(i) : 0;
    const charB = i < b.length ? b.charCodeAt(i) : 0;
    result |= charA ^ charB;
  }
  // Include length check IN the constant-time result
  result |= (a.length === b.length) ? 0 : 1;
  return result === 0;
}
```

**Action Taken:**
- Reverted 16's commits: 5900df23, 52d06c29, ee796c62
- Verified commit 697215a7 contains the correct implementation

**Lesson:** Constant-time comparison means *constant time* — no early exits based on input properties.
