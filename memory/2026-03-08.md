## Security Fixes — Authentication Bypass Vulnerability

### Timeline

**Commit `f13d0010` (Original attempt):**
- Fix: `if (providedKey && !timingSafeEqual(providedKey, expectedKey || ''))`
- Issue: Only validates when client provides a key; skips check if client omits `X-API-Key` header
- Result: **Authentication bypass vulnerability** when server has `API_KEY` configured but client doesn't send header

**Commit `d370c497` (Correct fix):**
- Fix: `if (expectedKey ? !timingSafeEqual(providedKey || '', expectedKey) : providedKey)`
- Logic: Always validate when server has key configured; reject if client provides key when server has none

### Authentication Matrix

| Server Config | Client Header | `f13d0010` Result | `d370c497` Result |
|--------------|---------------|-------------------|-------------------|
| `API_KEY=secret` | `X-API-Key: secret` | ✅ Pass | ✅ Pass |
| `API_KEY=secret` | (no header) | ❌ **PASS** (BYPASS) | ❌ Fail ✅ |
| (no key) | (no header) | ✅ Pass | ✅ Pass |
| (no key) | `X-API-Key: anything` | ✅ Pass | ❌ Fail ✅ |

# Security Notes — TimingSafeEqual Implementation (March 2026)

## Problem Statement
- JavaScript's `a === b` fails on equal-length mismatch due to short-circuit early exit
- `Math.max(a.length, b.length)` prevents length oracle but leaks difference via XOR timing

## Recommended Implementation
- Use `crypto.timingSafeEqual(a, b)` where available (Node.js)
- For n8n Code nodes (no crypto module), implement best-effort:
  - Fixed-length comparison (256 chars max, padded with `\0`)
  - Early exit on length mismatch (acceptable for constant-length secrets)
  - XOR comparison for fixed-length portion

## Current Status
- n8n implementation uses fixed-length comparison (256 chars max)
- Prevents timing attacks on API key validation
- **Constraint:** n8n Code nodes don't have access to Node.js crypto module
- **Status:** Best-effort implementation within n8n sandbox constraints
