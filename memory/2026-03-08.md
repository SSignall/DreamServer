## 2026-03-08 Late — Whisper STT Security Vulnerability (Android-18 Review)

**Date:** 2026-03-08 05:21 EDT  
**Commit:** `c9b498a5`  
**Review:** Android-18

### Issues Found

**[CRITICAL] Incomplete diff** — The `pathTraversalPattern.test(normalizedPath)` line is cut off mid-statement, causing a syntax error (missing closing `rn` and `}`), which breaks the workflow.

**[MEDIUM]** Removed fallback to `$input.first().json.api_key` in `providedKey` assignment — previously supported both header and body `api_key`, now only checks header. This is a breaking change in API contract.

**[HIGH]** The `timingSafeEqual` function no longer checks `aLen === bLen`, relying solely on `result === 0`. While padding mitigates length leaks, omitting explicit length check is a subtle regression — an attacker could potentially exploit if `expectedKey` is empty and `providedKey` is a prefix match under padding.

**[LOW]** `MAX_LEN = 256` is arbitrary; `WHISPER_API_KEY` is likely short — consider aligning with typical JWT/API key lengths (e.g., 64–128) to avoid unnecessary CPU use, though not critical.

### Fix Applied (Commit `1819b40b`)

- Restored body fallback: `providedKey = (apiKeyHeader ? headers[apiKeyHeader] : null) || $input.first().json.api_key`
- Added explicit length check: `return result === 0 && a.length === b.length`
- Changed from truncation+padding to `Math.max(a.length, b.length)` iteration with out-of-bounds = 0
- Removed `MAX_LEN` truncation — keys can be any length (only limited by `Math.max`)

**Status:** Fix committed and pushed to `dev/main`.

## 2026-03-08 Late — LocalAI Security Vulnerabilities (Android-18 Review)

**Date:** 2026-03-08 05:31 EDT  
**Commit:** `99b2c2d3`  
**Review:** Android-18

### Critical Issues

**[CRITICAL] Prefix attack vulnerability** — `timingSafeEqual()` truncates inputs to 256 chars via `MAX_LEN = 256`, allowing authentication with only the first 256 characters of a longer `LOCALAI_API_KEY`. An attacker can authenticate using a partial key match.

**[HIGH] Breaking change** — Removes support for API key in request body (`$input.first().json.api_key`). Clients sending keys via JSON payload will fail authentication. Only header-based auth (`x-api-key`) is now supported without a deprecation path.

**[MEDIUM] Suffix ignored** — If provided API key exceeds 256 characters but matches expected key's first 256 chars, authentication incorrectly succeeds (suffix ignored), allowing key suffix tampering.

### Fix Required

1. Remove the 256-character truncation (`MAX_LEN`) to allow full-length key comparison
2. Restore body-based API key support (`$input.first().json.api_key`) as fallback
3. Use `timingSafeEqual` with proper length checking (early exit on mismatched lengths)

### Root Cause

The `timingSafeEqual` function pads both strings to `MAX_LEN = 256` and compares only those 256 characters, ignoring any characters beyond that. This creates a partial-key vulnerability.
