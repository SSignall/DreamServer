## Issue #33 - Dashboard Offline (CachyOS/AMD) — Partial Resolution

**Problem:** CachyOS user with AMD Strix - containers healthy but dashboard reports all services offline.

**Root Cause:** Docker DNS resolution failing on CachyOS - containers on same network but cannot resolve service names like `dashboard-api`.

**First Fix Attempt (Commit `ea3d1753`):**
Added explicit DNS servers to both `dashboard` and `dashboard-api` services:
```yaml
dns:
  - 127.0.0.11
  - 8.8.8.8
  - 1.1.1.1
```

**Security Review (Android-18):** Hardcoded external DNS (8.8.8.8, 1.1.1.1) bypasses Docker daemon DNS, creates external dependencies, and risks leaking internal hostname queries.

**Final Fix (Applied by Android-17):** Removed public DNS resolvers, kept only Docker's internal DNS:
```yaml
dns:
  - 127.0.0.11
```

**Why This Works:** Docker's internal DNS (`127.0.0.11`) handles both internal service resolution (e.g., `llama-server`, `qdrant`) and external lookups via the host's resolver configuration — no leaks, no hardcoded dependencies.

**Status:** ✅ Fixed and pushed to `dev/main`

## Key Learnings

### Docker Compose Build Context
When running from repo root with `-f extensions/services/X/compose.yaml`:
- `context: extensions/services/X` correctly resolves to the service directory
- `context: .` fails because Docker looks for Dockerfile relative to compose file location

### DNS Configuration Best Practice
Use only Docker's internal DNS (`127.0.0.11`) to avoid:
- Leaking internal hostname queries to public resolvers
- Creating external network dependencies
- Bypassing host DNS configuration

### Review Workflow
Android-18's security reviews catch issues that initial implementations miss:
- Hardcoded external DNS leaks
- Build context path resolution edge cases
- Missing healthchecks/resource limits (false positives)
