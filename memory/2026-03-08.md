**Issues Found & Fixed:**

### [HIGH] Missing "Check Valid" IF Nodes
The Ollama and RVC workflows returned `valid: false` on validation failure, but **lacked the IF node to branch on that result**. Validation failures continued to API calls with invalid data.

**Fix:** Added "Check Valid" IF nodes to both workflows that route to error responses when `valid === false`.

### [MEDIUM] timingSafeEqual Timing Side-Channel
Previous implementation had early return on length mismatch that leaked timing info:
```javascript
// VULNERABLE (early return leaks which input was longer)
if (a.length > MAX_LEN || b.length > MAX_LEN) return false;
```

**Fix:** Eliminated early returns. Always pad both inputs to MAX_LEN and compare full length.

### [MEDIUM] Missing Type Validation
Some string operations lacked type checks.

**Fix:** Added `typeof === 'string'` validation before all string operations on model, message, system, channel inputs.

### [LOW] Duplicate MAX_LEN Declarations
Multiple `const MAX_LEN = 128` in same scope.

**Fix:** Removed duplicates, single declaration at top of each validation function.

# Security Notes — TimingSafeEqual Implementation (March 2026)

## Problem Statement
- JavaScript's `a === b` fails on equal-length mismatch due to short-circuit early exit
- `Math.max(a.length, b.length)` prevents length oracle but leaks difference via XOR timing

## Recommended Implementation
- Use `crypto.timingSafeEqual(a, b)` where available (Node.js)
- For n8n Code nodes (no crypto module), implement best-effort:
  - Fixed-length comparison (256 chars max, padded with `\0`)
  - Early exit on length mismatch (acceptable for constant-length secrets)
  - XOR comparison for fixed-length portion

## Current Status
- n8n implementation uses fixed-length comparison (256 chars max)
- Prevents timing attacks on API key validation
- **Constraint:** n8n Code nodes don't have access to Node.js crypto module
- **Status:** Best-effort implementation within n8n sandbox constraints

## Verification
- JSON validation passes
- All HIGH issues from 18's review resolved
- No insecure fallbacks to user input

## 2026-03-08 — timingSafeEqual Regression Fix

**Android-18 Alert:** Flagged security pattern regressions in `d7786cbf`

**Root Cause:** My commit `d74a7c2f` incorrectly standardized timingSafeEqual to the **flawed** Pattern B (MAX_LEN=256 with `result |= length check`) instead of the secure Pattern A (MAX_LEN=128 with `padEnd`).

**Impact:** RVC and Ollama workflows used the less secure implementation. Bark was already correct.

**Fix Applied:**
- Changed `MAX_LEN=256` → `MAX_LEN=128` 
- Removed `result |= (a.length === b.length) ? 0 : 1` pattern
- Added `padEnd(MAX_LEN, '\0')` for true constant-time comparison
