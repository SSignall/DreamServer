#### GPU Count Negative Value Rejection (Commit `77d77fbe`)

**Issue (K2.5 Review of `699a4126`):**
- Exception-based flow allows negative GPU counts (e.g., `gpu_count_val = "-1"`) to pass through `int()`
- Results in `gpu_count = -1` instead of falling back to default (0 or 1)

**Fix Applied:**
```python
try:
    gpu_count = int(gpu_count_val) if gpu_count_val is not None else None
    if gpu_count < 0:
        gpu_count = None
except (ValueError, TypeError):
    gpu_count = None
gpu_count = gpu_count if gpu_count is not None else (1 if gpu_type not in {"none", ""} else 0)
```

**Validation:** Added explicit `if gpu_count < 0: gpu_count = None` check before fallback

**Test Results:**
- `count=0` → returns `0` (explicit 0 preserved via `is not None` check)
- `count='N/A'` → returns `1` (non-numeric falls back to default)
- `count='-1'` → returns `1` (negative falls back to default)
- `count=None` → returns `1` (missing field falls back to default)

### Final State (980dceaf)

**Architecture**: Simplified 4-node workflow
1. **Webhook Trigger** → receives JSON input
2. **Bark API Call** → makes HTTP request to Bark service
3. **Process Response** → merges Bark response with original input via `$input.all()`
4. **Discord Output** → success notification

**Error Flow**: Bark API Call error → Discord Error

**Key Implementation Details**:
- Process Response gets Input 0 (Bark API) and Input 1 (Webhook Trigger payload)
- Discord Error uses `$('Webhook Trigger').first().json.channel` for channel access
- Error truncation: `.toString().slice(0, 1000)` on message, `.slice(0, 100)` on status
- Environment variables: `BARK_HOST`, `BARK_EXTERNAL_PORT` with localhost:9200 defaults

**Quality Checks**:
- ✅ Valid JSON syntax
- ✅ Proper n8n expression syntax (`={{ }}` not Jinja `{{ }}`)
- ✅ Error truncation prevents Discord 2000-char limit failures
- ✅ Node comments document input format and env vars
- ✅ Connections properly route both success and error paths

### Coaching Notes for 16

The iteration chain shows a common pattern:
1. Initial fix introduced Store Input + Merge nodes to solve data persistence
2. Merge node used `mode: "combine"` with empty `mergeByFields` - this doesn't work as expected
3. Better solution: Remove intermediate nodes, use n8n's built-in multi-input routing
4. Final design is cleaner: Webhook feeds both Bark API (index 0) and Process Response (index 1), Bark API also feeds Process Response (index 0)

**Pattern to remember**: When you need access to original input in a downstream node, connect the trigger directly to that node on a secondary input index. Avoid Merge nodes unless actually merging disparate data streams.
