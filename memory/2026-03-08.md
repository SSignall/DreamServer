## 2026-03-08 Late — LocalAI Security Vulnerabilities (Android-18 Review)

**Date:** 2026-03-08 05:31 EDT  
**Commit:** `99b2c2d3`  
**Review:** Android-18

### Critical Issues

**[CRITICAL] Prefix attack vulnerability** — `timingSafeEqual()` truncates inputs to 256 chars via `MAX_LEN = 256`, allowing authentication with only the first 256 characters of a longer `LOCALAI_API_KEY`. An attacker can authenticate using a partial key match.

**[HIGH] Breaking change** — Removes support for API key in request body (`$input.first().json.api_key`). Clients sending keys via JSON payload will fail authentication. Only header-based auth (`x-api-key`) is now supported without a deprecation path.

**[MEDIUM] Suffix ignored** — If provided API key exceeds 256 characters but matches expected key's first 256 chars, authentication incorrectly succeeds (suffix ignored), allowing key suffix tampering.

### Fix Required

1. Remove the 256-character truncation (`MAX_LEN`) to allow full-length key comparison
2. Restore body-based API key support (`$input.first().json.api_key`) as fallback
3. Use `timingSafeEqual` with proper length checking (early exit on mismatched lengths)

### Root Cause

The `timingSafeEqual` function pads both strings to `MAX_LEN = 256` and compares only those 256 characters, ignoring any characters beyond that. This creates a partial-key vulnerability.
